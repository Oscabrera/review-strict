# Changelog

## 1.2.0
- **Cost lever ŌĆö hybrid per-lens model.** `/review-strict` (5 lenses) and `/spec-strict` (6 lenses) now split their fan-out across model tiers instead of putting every lens on the session model:
  - **Deep-reasoning lenses run on the session model** (review: correctness, security, architecture; spec: coverage, risk, architecture, scope) ŌĆö these are where a missed subtle bug (a **false negative**) is most costly, so they keep the strong model.
  - **Mechanical lenses run on Sonnet** (review: tests, migration; spec: ac-quality, verification) ŌĆö tautological-test / two-phase-migration / wrong-stack-command / filler-table checks are checklist-style, where Sonnet is enough.
  - **Rigor gate untouched:** the adversarial `verify-skeptic` pass and synthesis **always** run on the session model, in every mode.
  - New `--model <sonnet|opus|haiku|inherit>` flag + `REVIEW_STRICT_MODEL` / `SPEC_STRICT_MODEL` env vars **force a uniform model for all lenses**, overriding the hybrid default: `--model sonnet` = cheapest (bulk runs), `--model inherit`/`opus` = maximum depth (every lens on the session model). Precedence: flag ŌåÆ env ŌåÆ hybrid.
- **`/spec-strict` graphify-first grounding.** Phase-1 grounding now queries `graphify-out/` (when present) before exhaustive `rg` to find callers/subclasses/layering ŌĆö a scoped subgraph means fewer file reads for the same proof. Falls back to `rg`/glob when there's no graph; grounding still ends in real `file:line` facts.
- Docs: the hybrid split + `--model` documented in all skill READMEs (en+es) and the root config table (all three model env vars side by side); "Bounded cost" operating rules name the split, `--model sonnet`, and `--fast` as the cost levers.

## 1.1.0
- New sibling skill **`/audit-strict`** ŌĆö deep, repo-adaptive whole-repository architecture audit (the constructive counterpart to the diff-scoped `/review-strict`).
- Reverse-engineers the system into a multi-file `audit-strict/` deliverable: component map, end-to-end flows (`sequenceDiagram`), entity lifecycle (`stateDiagram-v2`), derived data model (`erDiagram`), quality audit, and a prioritized Fase 0ŌĆō5 roadmap + `analisis.json`.
- Five read-only cartographer agents (`cart-components`, `cart-flows`, `cart-lifecycle`, `cart-datamodel`, `cart-quality`) fanned out one-per-deliverable for even depth.
- Reuses review-strict's spine: repo profile (Phase 0), `verify-skeptic` adversarial pass (Phase 3), severity crosswalk, and the staff baseline.
- Cite-accuracy guarantees: every `file:line` survives the skeptic pass; no `{{placeholder}}` leaks; no "X is missing" claim that contradicts the repo profile; `analisis.json` validated with `jq`. Traceability applies everywhere including the README summary; N+1/in-loop findings cite the loop line AND the per-iteration query line.
- Balanced summary: the README uses a per-dimension scorecard (1ŌĆō5) + findings grouped by dimension, so the dimension holding the P0/P1s (usually security) can't monopolize visibility; global priority ordering lives in `recomendaciones.md`.
- Coverage rules: `cart-flows` must trace the primary user-facing/product flow (not only webhook/async flows); `cart-quality` audits per-resource ownership (IDOR) explicitly ŌĆö missing-ownership and ownership-from-unverified-input are findings distinct from missing-authentication.
- Decision-usefulness upgrades (precision ŌåÆ risk): (1) **Confianza y condicionalidad** box leads the README ŌĆö "verified" = citation accuracy, NOT exploitability; security severities are conditional on reach with a priority-`0` "confirm exposure" gate. (2) New **Dependencias & CVEs** dimension grounded in Phase 1 (framework EOL, `composer/npm audit` best-effort, dep-bot) ŌĆö unscanned CVEs are an explicit gap, not "clean". (3) New **Phase 3.5 completeness-critic** emits a traced/shallow/not-audited **coverage map** and catches missed product-core flows before synthesis. (4) Finding contract gains a `reach` field (who can trigger + precondition) so gravity is separated from likelihood. (5) Active harm vs latent/aspirational debt is separated ŌĆö a declared-but-unadopted standard is P2/P3, not P1.
- Auto-detects backend (Laravel/PHP) vs Vue/Nuxt and selects the cartographer variant.
- Cost control: the five read-heavy cartographers run on **sonnet by default** (`--model` / `AUDIT_STRICT_MODEL` to change; `inherit` uses the session model), while the verify pass and synthesis stay on the session model so rigor is unaffected. Whole-repo audits are read-intensive, so cheaper readers are the biggest cost lever; graphify-first grounding cuts reads further when `graphify-out/` exists.
- Configurable output folder and language, per-developer or per-run. Folder: `--out` / `AUDIT_STRICT_OUT_DIR` (falls back to shared `REVIEW_STRICT_ARCHIVE_DIR`); a configured base is grouped by tool then repo to `<base>/audit-strict/<repo>/` (no collision with review-strict reports), else in-repo `<repo-root>/audit-strict/`. Language: `--lang` / `AUDIT_STRICT_LANG` (falls back to shared `REVIEW_STRICT_LANG`, else `en`).
- **`/review-strict` architecture lens upgraded** ŌĆö catches the "PR con toda la l├│gica en un archivo" case (SRP/monolith, escalated), SOLID and design-pattern conformance **to the repo's own patterns** (evidenced only, never architecture-astronaut), and concrete complexity/size budgets inspired by PHPMD + PHP Insights (cyclomatic complexity, WMC, method/class length, params, coupling). Repo config (`phpmd.xml`/`phpinsights.php`/PHPStan) wins on thresholds; since those tools rarely run in CI here, the lens estimates from the diff itself (exact counts + CC as decision-point estimate) rather than depending on them.
- **New sibling skill `/spec-strict`** ŌĆö pre-code review of a Stratos/Axiom spec dir (`spec.md`/`plan.json`/`pr.md`/`validation.md`) before Forge writes code. Six spec-native lenses grounded in the codebase and adversarially verified: entry-point/caller coverage, AC diff-checkability, verification-command sanity vs the real stack, grounded blocker/risk inventory, architecture/layering fit, scope completeness. Emits a readiness verdict (`ready` / `revise-before-Forge`) + findings phrased as concrete spec edits; read-only (never edits the spec). Completes the family: `/spec-strict` (the plan) ŌåÆ `/review-strict` (the change) ŌåÆ `/audit-strict` (the repo).

## 1.0.0
